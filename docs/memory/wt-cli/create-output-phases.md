# wt-cli: Create Output Phases Contract

> Post-implementation behavior capture for the `wt create` / `wt init` phase-separator output
> and the CLI's stdout/stderr stream discipline.
> Source changes: `260531-pnmi`, `260622-log5`, `260626-n6ma`, `260717-aeka`, `260717-ohwb`
> (each contract below carries its citation).

This file documents the phase-separator output contract that `wt create` and `wt init` honor, and — as the canonical stdout/stderr stream-discipline file — the single `wt.Warn` warning emitter that all `Warning:` call sites route through. Future changes touching `src/cmd/wt/create.go`, `src/cmd/wt/init.go`, `src/cmd/wt/delete.go`, or `src/internal/worktree/{crud.go,errors.go}` should preserve these invariants unless an explicit spec amendment supersedes them.

## Requirements

### `PhaseSeparator` is the sole separator producer

- `src/internal/worktree/errors.go` exposes `func PhaseSeparator(label string) string`, alongside the other output helpers (`WtError`, `PrintError`, `PrintInitFailureBanner`). It is the **single** source of phase-separator strings — call sites MUST NOT reinline the glyph/width/color logic.
- The returned string has **no trailing newline**; callers add it via the `fmt.Fprintln(os.Stderr, ...)` they already use.
- Fixed total **visible** width of `phaseSeparatorWidth = 40` columns (label + glyphs + spaces). It MUST NOT query the terminal size — fixed width keeps output deterministic for tests and matches the single-binary / no-hidden-state posture (Constitution I).
- Visible width is **rune-counted** (`utf8.RuneCountInString(label)`), not byte-counted, so a label with multi-byte runes (e.g. a non-ASCII `WORKTREE_INIT_SCRIPT` path) still renders at the fixed visible width.
- Layout: two leading glyphs, a space, the label, a space, then enough trailing glyphs to fill 40 visible columns. ANSI escapes around the label are NOT counted toward the width. Trailing glyph count is clamped at 0 for labels at or over the budget.
- Color detection uses the **blanked package-level color vars** (`colorEnabled := ColorReset != ""`), reusing the existing `init()` that blanks `ColorBold`/`ColorReset`/etc. when `NO_COLOR` is non-empty. It MUST NOT call `os.Getenv` afresh.
  - Color enabled: unicode rule `── Label ──…` (glyph `─`, U+2500 BOX DRAWINGS LIGHT HORIZONTAL) framing a `ColorBold`-wrapped label.
  - `NO_COLOR` set (color vars blanked): plain-ASCII rule `-- Label --…` (glyph `-`) with **no** ANSI escape sequences.

### `wt create` emits Git, Init, and Open separators on stderr, in order

- `wt create` emits up to three phase separators to **stderr** (never stdout), in order, each immediately preceding the output of its phase:
  1. `PhaseSeparator("Git")` — emitted in `create.go` immediately before the deferred summary block (`Created worktree:` / `Path:` / `Branch:` / `From:` — see "The Git-phase summary block carries a fourth `From:` line" below), joining the deferred-summary emission under the rollback handler.
  2. The **Init** separator — emitted by the init runner, NOT by `create.go` (see next requirement).
  3. `PhaseSeparator("Open")` — emitted immediately before the open phase. When the init phase ran, the Open separator + menu render are preceded by an **unconditional foreground reclaim** (`reclaimTerminalForeground(ttyFd, wtPgid)` in `create.go`, just before the Open separator, gated on the same `term.IsTerminal` check as capture) so the Open phase can never SIGTTOU on a shared-TTY init child that stranded terminal foreground (260602-z4p7). The reclaim is a job-control ordering guarantee, not an output change. See `init-failure-contract.md` "Terminal foreground is reclaimed after the init child".
- A separator is emitted **only when its phase produces output** — a separator never precedes a phase that emits nothing:
  - **Git** is emitted on every successful create, because the deferred summary block always prints (it is the Git phase's output).
  - **Init** is emitted only when the init phase actually runs an init command — NOT when init is disabled and NOT on the `*InitNotFound` path.
  - **Open** is emitted only when the open phase runs, gated on `worktreeOpen != "skip"` (the internal variable, bound by both `--open` and its deprecated alias `--worktree-open`). `--open=skip` (and `--non-interactive`, which defaults open to `skip`) suppress it. The Open phase is **also reachable after an init failure** on the interactive open-anyway *Yes* path (260626-n6ma). On the open-anyway *No* path the failure branch sets `worktreeOpen = "skip"`, so the separator + menu are suppressed there. The stdout discipline holds (see below): on any init-failure path the function exits `ExitInitFailed = 7` via the end-of-function `initFailed` guard, which sits **before** the stdout path-line print, so the path line is NOT emitted on an init-failure path (even after a successful open-anyway open).
- The Git separator MUST stay inside the existing deferred-summary emission, **before** the init-phase `signal.Reset(syscall.SIGINT, syscall.SIGTERM)` call. It MUST NOT introduce new I/O or a prompt inside the tight reinstall window between `git worktree add` returning and that `signal.Reset` (see `init-failure-contract.md` "SIGINT during init").
- The summary/warning/prompt substrings `Created worktree:`, `Open in:`, and the `Path:` / `Branch:` summary lines are load-bearing test anchors. The `From:` line is the fourth line of the same `Fprintf` (260717-aeka).

### The Git-phase summary block carries a fourth `From:` line

- The deferred Git-phase summary `fmt.Fprintf(os.Stderr, ...)` emits **four** `key: value` lines — `Created worktree:` / `Path:` / `Branch:` / `From:` — in that order, all in the **same** Fprintf call under the rollback handler (on **stderr**, inside the reinstall-window discipline above; stdout stays the single path line) (260717-aeka).
- The `From:` value names *which branch-selection mode* ran (the modes are defined in `create-branch-semantics.md`) and *what ref* the branch was based on, so the create-vs-checkout path is visible at creation time:
  - **positional new branch / bare exploratory, `--base` given** → the `--base` ref **verbatim as typed** (no `git rev-parse` resolution — the user's own token is the label).
  - **positional new branch / bare exploratory, no `--base`** → the resolved HEAD label from `wt.DescribeHead()` (current branch name, or the short SHA when detached).
  - **`--checkout <branch>`** → the fixed copy `existing branch '<branch>' (checked out directly)` (naming the existing-branch path; the remote-only fetch-then-checkout variant uses the **same** copy — the user-visible outcome, a worktree sitting directly on the existing branch, is identical).
- The `From:` value is resolved into a `createdSummaryFrom` local **BEFORE** the create-dispatch `switch` (before any `git worktree add`), mirroring the existing `createdSummaryBranch` idiom. This is load-bearing: the only git query the value may perform (`wt.DescribeHead()`, taken only in the no-`--base`/no-`--checkout` case) MUST NOT enter the tight reinstall window between `git worktree add` returning and the init-phase `signal.Reset`. Resolving pre-dispatch is equivalent — `wt create` never moves the invoking worktree's HEAD, so HEAD at dispatch equals HEAD at summary time.
- `wt.DescribeHead()` (in `src/internal/worktree/context.go`) is **best-effort**: it returns the branch name (`git rev-parse --abbrev-ref HEAD`), or the short SHA (`git rev-parse --short HEAD`) when abbrev-ref returns the literal `HEAD` (detached), and falls back to the string `"HEAD"` on any git error or empty output. It returns no error and can never fail or abort the create — the `From:` line is informational (Constitution V keeps the git op in `internal/`, not inlined in `cmd/`).
- The `--reuse` path emits **no** `From:` line — it prints `Reusing existing worktree:` and no Git-phase summary block at all (no branch created or checked out, nothing to show). See `create-branch-semantics.md` "`--reuse` is unchanged".

### Init separator is owned by the runner, labeled with the resolved command

- The Init separator is emitted by the code that owns the `Running worktree init...` line, so it is produced **exactly once** regardless of caller — a single emission point per path:
  - `RunWorktreeSetupWithObserver` in `src/internal/worktree/crud.go` (the `wt create` init step) — `crud.go:157`.
  - `runInitScript` in `src/cmd/wt/init.go` (the standalone `wt init` path) — `init.go:78`.
- The label is `Init (<cmd>)`, where `<cmd>` is the resolved init command as the user would recognize it (e.g. `Init (fab sync)`, or `Init (scripts/setup.sh)` for a file-path invocation). It is derived from the same init-script value resolution uses (`InitScriptPath()` / the resolved `*exec.Cmd`) and is **never hardcoded**.
- On the `*InitNotFound` path, the existing `RenderWarning()` is used unchanged and **no** Init separator is emitted — there is no command to label. The graceful-skip exit-0 behavior is unchanged (see `init-failure-contract.md`).
- The substrings `Running worktree init` and `Worktree init complete` remain present — the separator augments these markers, it does not replace them (preserves existing test assertions).

### STDOUT discipline: separators are stderr-only

- All phase separators are written to **stderr**. None ever appears on stdout.
- `wt create`'s stdout remains **solely** the final worktree-path line (`fmt.Println(wtPath)`), byte-identical to before this change, **on the success path**. This preserves the launcher-contract guarantee that stdout is the machine-readable result. No separator, summary, banner, prompt, or init output leaks to stdout.
- **On any init-failure path the path line is NOT printed** (`260626-n6ma`). The end-of-function guard `if initFailed { os.Exit(wt.ExitInitFailed) }` (`create.go:476–478`) sits **before** `fmt.Println(wtPath)` (`create.go:482`), so a non-zero init exit — including after a successful interactive open-anyway open — exits 7 with **no** stdout path line. This is the correct stdout discipline: stdout is the success result, and an init failure is not a success. Operators detect the kept worktree via exit code 7 (and the stderr banner's absolute `Worktree:` path / `Go:` hint), not via a stdout line. The init-failure banner and the open-anyway prompt are stderr writes, consistent with stdout=machine-result / stderr=human.
- ALL of `wt init`'s init diagnostics go to stderr: the `Running worktree init...` / `Worktree init complete.` banner (`fmt.Fprintln(os.Stderr, ...)`), the Init separator, AND the init script's own stdout (the init child is wired with `cmd.Stdout = os.Stderr` at `init.go:83`, joining `cmd.Stderr = os.Stderr`). `wt init` has **no machine-readable stdout result** — it is a side-effecting command whose outcome is its exit code — so all of its output is diagnostic.
- `wt init` exits `ExitInitFailed = 7` on a script non-zero exit, with one carve-out (260705-irnt): the **built-in default** `fab sync` exiting fab-kit's `ExitNotManaged = 3` in a non-fab repo is classified (`DefaultNotApplicable`) as "default does not apply" — `wt init` writes the skip warning and exits 0 instead of 7 (an explicit `WORKTREE_INIT_SCRIPT` value, or any other exit code, still exits 7; see `init-failure-contract.md`). `wt init` tests assert on the combined `r.Stdout + r.Stderr`.

### `wt.Warn` is the one-line warning helper; warnings are stderr-only

- `src/internal/worktree/errors.go` exposes `func Warn(format string, args ...any)` alongside the other output helpers (`WtError`, `PrintError`, `PhaseSeparator`, `PrintInitFailureBanner`) (260622-log5). It is the **shared one-line** `Warning:`-diagnostic helper used by `create`/`delete` — it writes `"%sWarning:%s %s\n"` (`ColorYellow` prefix, `ColorReset`, then the formatted message) to **stderr**. (Exceptions: the two multi-line `Warning:` renderers in `internal/worktree/init.go` — `InitNotFound.RenderWarning` (verbose not-found copy) and `RenderDefaultSkipWarning` (the `260705-irnt` default-not-applicable skip copy) — build their own multi-line text and do **not** route through `Warn`, so `Warn` is not the CLI's sole `Warning:` producer.)
- It is the warnings analogue of `PhaseSeparator`: color detection reuses the **blanked package-level color vars** (the `NO_COLOR`-blanking `init()` zeroes `ColorYellow`/`ColorReset`), so under `NO_COLOR` it emits a plain `Warning: <msg>\n` with **no** ANSI escapes. It MUST NOT call `os.Getenv` afresh.
- **All one-line `Warning:` call sites route through `Warn`** — `create.go` (the uncommitted-changes warning and the open-phase "could not open in …" / reused-worktree-init warnings) and `delete.go` (the two `failed to remove …` warnings and the two pre-menu warnings). The two multi-line renderers noted above (`InitNotFound.RenderWarning` and `RenderDefaultSkipWarning`, both in `init.go`) stay out of band — `Warn` owns the *one-line* `Warning:` idiom, not the multi-line ones. Call sites pass the **message text only**, never the literal `Warning: ` prefix — the helper owns the prefix, so re-including it would double it.
- **`wt delete`'s two pre-menu warnings route through `wt.Warn` (stderr).** The uncommitted-changes warning (`handleUncommittedChanges`) and the unpushed-commits warning (`handleUnpushedCommits`) route through `wt.Warn`, matching the sibling warnings on stderr in the same file (260622-log5). Warnings are diagnostics and belong on stderr.
- **Menu-layout spacing lives at the call site, not in the helper.** `Warn` appends exactly one trailing `\n`. The two pre-menu warnings need a blank line before and after for menu spacing, so each call site frames the `wt.Warn(...)` with a bare `fmt.Fprintln(os.Stderr)` immediately before and after (the `\n…\n\n` framing).
- The helper owns only the prefix, stream, and color — never the message text.

### `wt delete`'s entire non-error human copy is on stderr; stdout is empty on every live path

- **ALL** non-error human copy in `src/cmd/wt/delete.go` (~60 call sites across nine handlers plus the SIGINT-handler framing newline) writes to **stderr** via `fmt.Fprintf(os.Stderr, ...)` / `fmt.Fprintln(os.Stderr, ...)` (260717-ohwb). This satisfies toolkit principle №2 (*stdout = data, stderr = diagnostics*, a MUST) for `wt delete` — see [toolkit-standards-conformance](/wt-cli/toolkit-standards-conformance.md) № 2.
- **Covered copy, by handler**: the SIGINT-handler framing `fmt.Fprintln(os.Stderr)` (`deleteCmd` RunE, ~line 69); `handleDeleteCurrent` / `handleDeleteByName` / `handleDeleteMultiple` / `handleDeleteAll` (`Worktree:`/`Branch:`/`Path:` blocks, `Removing worktree...`, `Deleted worktree:`, `Cancelled.`, the multi-delete `Worktrees to delete (N):` / `Found N worktree(s):` summary blocks and per-item lines, the `--- Deleting: X ---` framing, `No worktrees found.`, and the current-worktree post-delete `You are no longer in a valid directory.` / `Run: cd <repo-root>` hint); `handleDeleteStale`'s `No idle worktrees (threshold: Nd).` empty state (see [idle-staleness-contract](/wt-cli/idle-staleness-contract.md)); `handleDeleteMenu`'s `No worktrees found.` / `Cancelled.`; `handleUncommittedChanges` / `handleStashInDir` stash-and-discard progress (`Stashing changes...`, `Changes stashed (hash: ...)`, `Discarding …`); `handleUnpushedCommits`'s `Commits that will be lost:` preview; and `handleBranchCleanup`'s `Deleted branch: %s (local/remote)` / `Skipped branch deletion: ...` / `Note: Could not delete remote branch`.
- **Direct `fmt.Fprintf(os.Stderr, ...)` idiom, no dedicated helper** — the same idiom `create.go` uses. Wording, argument order, color escapes (`wt.ColorBold`/`wt.ColorGreen`/`wt.ColorYellow`/`wt.ColorReset`), and blank-line framing are independent of the stream rule.
- **Empty stdout on every live path; the `--dry-run` preview is the stdout machine result.** `wt delete`'s stdout is **empty on every live delete code path**. Under `--dry-run`, the `Dry run — no changes will be made.` header and the `Would …` preview lines print to **stdout** — the preview *is* the machine result the caller asked for (260717-p5m9; see [delete-dry-run-contract](/wt-cli/delete-dry-run-contract.md)). This mirrors the stdout-is-machine-result discipline `wt create` keeps (its single path line) and `wt init` keeps (empty machine result); a future `--json` deleted-worktrees result can still claim the live path's empty stdout.
- **Shared menu rendering stays on stdout — a deliberate exception.** `wt.MenuSession.Show` → `showInteractive(os.Stdout, ...)` in `src/internal/worktree/menu.go` (and its non-TTY fallback) is shared infrastructure used by `create`/`open`/`go`/`delete` alike, with its own contract (`menu-navigation-contract.md`); flipping it is a cross-command change with its own scope, so the interactive delete menu's option rows render to **stdout**. The "empty stdout on every live path" statement above is about `delete.go`'s own copy — the menu it renders via the shared session is a separate surface.
- **Stderr-by-construction sites**: the two pre-menu warnings' `wt.Warn` + `fmt.Fprintln(os.Stderr)` framing, the two `wt.Warn("failed to remove …")` calls, and all `wt.ExitWithError` / `wt.PrintError` paths.

## Design Decisions

### Single helper in `errors.go` as the sole separator producer

`errors.go` already centralizes user-facing output formatting (`WtError`, `PrintInitFailureBanner`) and owns the `NO_COLOR`-blanking `init()`. Co-locating `PhaseSeparator` reuses that mechanism and keeps the canonical glyph/width/color wording in one place — mirroring how `RenderWarning` is the single source for not-found warnings. Inlining rule construction at each call site (`create.go`, `crud.go`, `init.go`) was rejected: it would duplicate the logic and invite drift.

### Init separator owned by the runner, not `create.go`

The runner already prints `Running worktree init...` and already holds the resolved `*exec.Cmd`, so it can label the separator with the resolved command and guarantee exactly one emission for both `wt create` and `wt init`. Emitting from `create.go` was rejected — it would either duplicate the resolution logic or double-label when `wt init` runs the same runner.

### `wt init` realigned to stderr (stream realignment in scope)

The convention captured by this change: **stdout = machine-readable result; stderr = diagnostics** (progress lines, banners, separators, prompts). `wt create` already followed it (stdout = the path line only). `wt init` used stdout for identical init diagnostics, an asymmetry that made separator placement inconsistent. Since `wt init` has no stdout contract (its tests assert on combined streams and no spec pins it to stdout), realigning it to stderr resolves the inconsistency and makes separator placement uniform across both commands. Keeping each command on its own stream was rejected — it enshrines the inconsistency.

### Single `wt.Warn` helper in `errors.go`, not a `cmd/`-local emitter

**Decision**: a single `Warn(format, args...)` lives in `internal/worktree/errors.go` next to `PhaseSeparator`/`WtError`/`PrintInitFailureBanner`, and every `Warning:` call site in `create.go`/`delete.go` routes through it; the documentation home is this stream-discipline file (a note here), **not** a new dedicated memory file.
**Why**: `errors.go` already centralizes user-facing output formatting and owns the `NO_COLOR`-blanking `init()`, so `Warn` reuses that mechanism and collapses ~10 divergent call sites to one idiom (DRY, and keeps `cmd/` thin per Constitution V) — exactly the rationale that put `PhaseSeparator` here. The warning realignment is the *same* stdout=machine / stderr=human convention this file already canonicalizes (see the `wt init` realignment above), so it belongs as a note here rather than fragmenting the stream-discipline story across two files. A standalone `warning-output-contract.md` would be the smallest file in the bundle for one helper + a two-call-site realignment, against the per-command / per-shared-concern granularity of the sibling files (none of `errors.go`'s other emitters got their own file).
**Rejected**: a `cmd/`-local warning helper (pushes formatting into the thin command layer, can't reuse the package color vars); a dedicated `warning-output-contract.md` memory file (over-granular for a copy-polish change; splits the one stream-discipline contract in two).
*Introduced by*: `260622-log5-dx-copy-polish`.

### `wt delete`'s whole non-error copy on stderr; menu left on stdout; stdout reserved

**Decision**: every non-error human-copy call site in `src/cmd/wt/delete.go` (all nine handlers plus the SIGINT-handler framing newline, ~60 sites) writes via `fmt.Fprint*(os.Stderr, ...)`, wording/colors/framing preserved; the shared menu rendering (`menu.go` `showInteractive` → stdout) is left as its own surface. `wt delete`'s stdout is empty on every live path. (The `260717-p5m9` `--dry-run` preview deliberately prints to stdout as the machine result — its `Dry run` header and `Would …` lines are exempt from the stderr rule; see [delete-dry-run-contract](/wt-cli/delete-dry-run-contract.md).)
**Why**: leaking progress/summary/hint copy to stdout was the one remaining principle-№2 gap the `260717-6end` toolkit-standards audit recorded. A mechanical stream flip at each site — reusing the `create.go` / log5 idiom, no new helper — is the smallest change that closes the gap; `wt delete` has no stdout machine contract, so nothing programmatic depends on the old stream, and an empty stdout is exactly what a future `--json` deleted-worktrees contract needs. The SIGINT framing newline is included so stdout stays empty even on the interrupt path. Keeping this as a note in *this* stream-discipline file rather than a new file keeps the one stdout=machine / stderr=human story in one place.
**Rejected**: introducing a shared `stderrPrintf` helper (a new abstraction the codebase does not use for this — the direct `fmt.Fprintf(os.Stderr, ...)` idiom is already the established pattern); flipping the shared menu rendering too (cross-command infrastructure with its own `menu-navigation-contract.md`, would churn `create`/`open`/`go` tests — its own backlog item); rewording any copy (out of scope — stream flip only); leaving the SIGINT framing newline on stdout (would keep stdout non-empty on the interrupt path).
*Introduced by*: `260717-ohwb-delete-copy-stderr-realign`.

### `From:` value resolved pre-dispatch into a local; `DescribeHead()` best-effort

**Decision**: `cmd/create.go` computes the `From:` summary value into a `createdSummaryFrom` local **before** the create-dispatch switch (`--checkout` → fixed copy; `--base` → verbatim; else → `wt.DescribeHead()`), and appends `From: %s\n` to the existing deferred-summary `Fprintf`. The HEAD-resolution helper `DescribeHead()` lives in `internal/worktree` and is best-effort (`"HEAD"` fallback, never errors).
**Why**: pre-dispatch resolution keeps the one possible git query out of the reinstall window (the same forbidden-I/O window the Git separator already respects), and HEAD cannot move between pre-dispatch and summary time since `wt create` never moves the invoking worktree's HEAD. Putting the git op in `internal/` (not inline in `cmd/`) follows Constitution V and mirrors the sibling `CurrentBranch()`; an informational line failing the create is the wrong tradeoff, so the helper swallows all git errors. Appending to the *same* Fprintf (no new output machinery) keeps the emission point single, exactly as the file's other separators do.
**Rejected**: resolving HEAD *inside* the deferred summary block (adds a subprocess call inside the forbidden reinstall window); returning `(string, error)` from `DescribeHead()` and threading the error into `cmd/` (a cosmetic label must not abort a successful create); resolving `--base` to a canonical SHA for display (the verbatim token is the most recognizable label — no clarity gain from a query).
*Introduced by*: `260717-aeka-create-summary-base-ref`.

### Fixed 40-column width, no terminal-size query

Deterministic output for unit tests; no dependency on terminal/ioctl; consistent with the single-binary, no-hidden-state posture (Constitution I). Dynamic width via terminal detection was rejected — non-deterministic in tests and adds a platform-specific code path for a cosmetic gain. (This mirrors `wt list`'s flag-based, no-`isatty` posture; see `list-status-contract.md`.)
