# Worktree Layout

Where worktrees live on disk, how they are named, and how the worktree
directory relates to its branch.

## Filesystem layout

Worktrees are created as **siblings** of the main repository, grouped under a
single per-repo directory:

```
<parent>/
├── <repo>/                  # main repo (clone or working directory)
└── <repo>.worktrees/        # all linked worktrees for this repo
    ├── swift-fox/
    ├── jolly-otter/
    └── crimson-heron/
```

The location is derived in `internal/worktree/context.go`:

```
WorktreesDir = filepath.Join(filepath.Dir(repoRoot), repoName + ".worktrees")
```

The main repo root itself is resolved by stripping `/.git` from
`git rev-parse --git-common-dir` — the same value regardless of whether the
command is run from the main repo or from a linked worktree.

## Naming convention

| Field | Value |
|-------|-------|
| **Pattern** | `{adjective}-{noun}` |
| **Example** | `swift-fox`, `jolly-otter` |
| **Generated by** | `wt create` (via `internal/worktree/names.go`) |
| **Override** | `--worktree-name <name>` |

`GenerateRandomName()` picks one entry from each of two curated word lists
(`adjectives` and `nouns`) and joins them with a single hyphen. The default
combinatorial space is large enough that collisions are rare.

`GenerateUniqueName(worktreesDir, maxRetries)` retries up to `maxRetries=10`
times to find a name that does not collide with an existing directory under
`<repo>.worktrees/`. After exhaustion, `wt create` exits with
`ExitRetryExhausted` and a message pointing the user at
`<repo>.worktrees/`.

When the user passes `--worktree-name`, no random name is generated and no
retry happens — the name is used as-is and an existing directory at that name
is an error (or, with `--reuse`, signals that the existing worktree should be
returned instead of created).

## Branch ↔ worktree relationship

The branch checked out in a worktree is **independent** of the worktree
directory name. Two illustrative cases:

1. **Exploratory worktree** (no positional `branch` arg): a new branch is
   created with the same name as the worktree (`swift-fox` worktree on
   `swift-fox` branch). The two strings happen to match, but they are
   independent values — renaming the branch later does not move the directory.
2. **Worktree for an existing branch** (positional `branch` arg given): the
   worktree directory still gets a random or user-supplied name, while the
   branch stays whatever the user passed (`<repo>.worktrees/swift-fox/`
   checked out on branch `feature/oauth`). When deriving a default worktree
   name from a branch, `DeriveWorktreeName` takes the last path segment after
   `/` and replaces non-alphanumeric characters (except `-` and `_`) with `-`
   — so branch `feature/oauth-pkce` yields suggestion `oauth-pkce`.

`wt delete --delete-branch auto` collapses these two cases by deleting the
branch only when it matches the worktree name — the typical exploratory case.
For the second case (worktree name and branch differ), `auto` keeps the
branch.
